# ConfigDeck

개발자가 프로젝트 시작 시 필요한 설정 파일(.eslintrc, .prettierrc, tsconfig 등)을 스택 기반으로 조합/생성/다운로드할 수 있는 다국어 웹 서비스이다.

## 기술 스택

| 영역            | 기술                             | 비고                                                           |
| --------------- | -------------------------------- | -------------------------------------------------------------- |
| 메타 프레임워크 | Astro                            | SSG 기반 정적 사이트. 콘텐츠 페이지는 JS 0KB                   |
| 인터랙티브 UI   | Svelte 5 (Runes)                 | Astro 아일랜드로 통합. 생성기 등 인터랙션이 필요한 부분에 사용 |
| 언어            | TypeScript                       | strict 모드 필수. any 금지, as 지양                            |
| 스타일링        | Tailwind CSS v4                  | 유틸리티 우선, @apply 지양, CSS 기반 @theme 설정               |
| 패키지 매니저   | pnpm                             | 엄격한 의존성 관리, store 기반 디스크 효율                     |
| 배포            | Cloudflare Pages                 | 정적 출력 배포, .dev 도메인, 무제한 대역폭                     |
| 린팅            | ESLint (flat config) + Prettier  | import 정렬: @ianvs/prettier-plugin-sort-imports               |
| 테스트          | Vitest (단위) + Playwright (E2E) | -                                                              |

## 하네스 구조

`.claude/` 디렉토리가 프로젝트의 기획, 의사결정, 컨벤션, 스킬의 중심 허브이다.
각 폴더에는 `index.md`가 있으며, 에이전트는 폴더 진입 시 반드시 `index.md`를 먼저 읽어야 한다.

```
.claude/
├── settings.json           # Claude Code 권한/자동화 설정
├── ia/                     # 기획 문서
│   ├── index.md
│   └── specs/
├── decisions/              # 의사결정 기록 (ADR)
│   ├── index.md
│   ├── template.md
│   └── records/
├── conventions/            # 코딩/스타일링/워크플로우 규칙
│   ├── index.md
│   ├── guides/
│   └── templates/
├── seo/                    # SEO/시맨틱 가이드라인
│   ├── index.md
│   └── guides/
├── research/               # 리서치 보고서
│   ├── index.md
│   └── reports/
├── hooks/                  # Hook 스크립트
├── skills/                 # 커스텀 스킬
│   ├── index.md
│   └── {skill-name}/
│       └── SKILL.md
├── agents/                 # 에이전트 정의
│   ├── index.md
│   └── {agent-name}.md
└── qa/                     # QA 하네스
    ├── index.md
    ├── templates/
    ├── guides/
    └── reports/
```

### 폴더별 목적

**`ia/`** — 기획 문서

- 목적: 서비스 기획서, IA, 페이지 구조, 사용자 플로우 등 기획 단계 문서를 보관한다. 에이전트가 기능 개발 시 서비스의 의도와 범위를 파악하는 데 사용한다.
- 배경: 프로젝트 시작 시 기획 문서를 코드와 함께 버전 관리하기 위해 도입

**`decisions/`** — 의사결정 기록 (ADR)

- 목적: 기술 스택, 아키텍처, 도구 선택 등 주요 의사결정을 ADR 형식으로 기록한다. 에이전트가 기존 결정의 맥락과 근거를 파악하여 일관된 방향으로 작업할 수 있도록 한다.
- 배경: 의사결정의 배경과 근거를 추적 가능하게 만들어, 재검토나 변경 시 판단 기준을 제공하기 위해 도입 (ADR-0001)

**`conventions/`** — 코딩/스타일링/워크플로우 규칙

- 목적: 코딩 규칙, 스타일링 규칙, 린팅 설정, 렌더링 전략, 브랜치/PR/커밋 워크플로우를 정의한다.
- 가이드: coding.md, styling.md, linting.md, rendering.md, workflow.md
- 배경: 여러 에이전트/세션에서 코드 스타일과 개발 프로세스가 일관되게 유지되도록 명시적 규칙을 제공하기 위해 도입

**`seo/`** — SEO/시맨틱 가이드라인

- 목적: 시맨틱 HTML 구조, 메타태그, OG 태그, 구조화 데이터(JSON-LD), 다국어 SEO 등의 가이드라인을 정의한다.
- 가이드: semantic-html.md, meta-tags.md, structured-data.md
- 배경: ConfigDeck은 SEO 유입이 핵심 전략(파일별/스택별 랜딩, 다국어 SEO 페이지)이므로, 별도 폴더로 분리하여 가이드라인의 비중을 높이기 위해 독립

**`research/`** — 리서치 보고서

- 목적: 기술 비교, UI/UX 레퍼런스, 라이브러리 선택 등 외부 조사 결과를 보관한다. ADR이나 기획서에서 근거 자료로 참조한다.
- 배경: 의사결정의 근거가 되는 리서치 자료를 기획(ia)이나 의사결정(decisions)과 독립적으로 관리하여, 참조 관계를 명확히 하기 위해 도입

**`hooks/`** — Hook 스크립트

- 목적: Claude Code의 자동화 Hook 중 복잡한 로직이 필요한 스크립트를 보관한다. 간단한 Hook은 settings.json에 인라인으로, 조건 분기가 있는 Hook은 `.sh` 파일로 분리한다.
- 배경: PostToolUse(자동 포맷팅), Stop(빌드/lint 검증) 등의 Hook을 통해 코드 품질을 자동으로 보장하기 위해 도입

**`skills/`** — 커스텀 스킬

- 목적: 반복 작업이나 특화 워크플로우를 SKILL.md 기반 커스텀 스킬로 정의한다. 슬래시 명령어(`/skill-name`)로 호출하거나 description 기반 자동 트리거로 활성화된다.
- 스킬 목록: research, create-pr, component-builder, lint-check, code-review, a11y-check, seo-audit, test-writer, e2e-test, biz-strategy
- 배경: 에이전트의 능력을 프로젝트 맥락에 맞게 확장하고, 반복 작업을 표준화하기 위해 도입

**`agents/`** — 에이전트 정의

- 목적: 프로젝트 전용 에이전트의 역할, 작업 원칙, 입출력 프로토콜을 정의한다. 에이전트 팀/서브 에이전트 아키텍처 패턴에 따라 구성된다.
- 개발 에이전트: config-maker, seo-specialist, ui-publisher, ux-designer, qa-agent
- QA 서브에이전트: unit-tester, e2e-tester, static-analyzer
- 비즈니스 분석 에이전트: market-intelligence, business-analyst, strategy-planner
- 배경: 복잡한 작업을 전문 에이전트에게 위임하고, 팀 단위 협업으로 품질을 높이기 위해 도입

**`qa/`** — QA 하네스

- 목적: QA 리포트 템플릿, 품질 게이트 기준, 테스트 가이드라인을 정의한다. qa-agent가 검증 결과를 종합하고 리포트를 생성할 때 참조한다.
- 구성: index.md (개요), templates/ (리포트 템플릿), guides/ (품질 게이트), reports/ (생성된 리포트)
- 배경: 단위 테스트, E2E 테스트, 정적 분석 결과를 표준화된 형식으로 종합하여 품질 게이트를 자동화하기 위해 도입

## 스킬과 에이전트의 역할 구분

스킬과 에이전트는 **서로 다른 레이어**에서 동작한다.

- **스킬 = 도구**: 한 가지 명확한 작업을 수행한다. `/seo-audit` → "이 페이지 SEO 검사해줘"
- **에이전트 = 전문가**: 특정 도메인의 종합적 전문성을 발휘한다. seo-specialist → "일본어 랜딩 페이지를 설계부터 구현까지 해줘"

|        | 스킬                                 | 에이전트                                         |
| ------ | ------------------------------------ | ------------------------------------------------ |
| 트리거 | `/스킬명` 수동 호출 또는 자동 트리거 | 복잡한 작업에서 서브에이전트로 위임              |
| 범위   | 단일 작업 (검사, 생성, 보고)         | 도메인 전체 (설계 + 구현 + 검증)                 |
| 예시   | `/lint-check` → Prettier/ESLint 검사 | qa-agent → 설정 유효성 + 빌드 + 테스트 통합 검증 |

스킬은 일상적인 반복 작업에, 에이전트는 복잡한 기능 구현에 활용한다. 동일 도메인에 스킬과 에이전트가 모두 있는 것은 의도적이며, 작업 규모에 따라 적절한 것을 선택한다.

### 에이전트 위임 필수 조건

다음 조건에 해당하는 요청은 **반드시 전문 에이전트에게 위임**해야 한다. 직접 처리하지 않는다.

| 요청 패턴                                | 위임 대상                      |
| ---------------------------------------- | ------------------------------ |
| "전반적으로 분석/검토해줘", "최적화해줘" | 해당 도메인 에이전트           |
| "설계해줘", "구조 잡아줘"                | ux-designer 또는 해당 에이전트 |
| "페이지/컴포넌트 구현해줘"               | ui-publisher                   |
| 설계 + 구현 + 검증이 결합된 작업         | 에이전트 파이프라인            |

**도메인별 에이전트 매핑:**

- SEO 관련 → `seo-specialist`
- 설정 스키마/옵션 관련 → `config-maker`
- UI/컴포넌트 구현 → `ui-publisher`
- UX/플로우 설계 → `ux-designer`
- 테스트/검증 → `qa-agent`
- 기획서 작성 → `product-planner`
- 시장 조사 → `market-intelligence`
- 경쟁력 분석 → `business-analyst`
- 전략 수립 → `strategy-planner`

> **Why:** 에이전트는 해당 도메인의 하네스 문서(가이드, 템플릿, 규칙)를 로드하고 전문성을 발휘하도록 설계되어 있다. 직접 처리하면 하네스를 참조하지 않아 품질이 저하되고, 도메인 지식이 분산되어 일관성이 깨진다.

### 토큰 최적화: 작업 규모별 도구 선택

스킬과 에이전트는 별도 컨텍스트 윈도우에서 실행되므로, 하네스 재로딩 + 웹 조사 결과 누적으로 토큰 소비가 크다. 작업 규모에 맞는 도구를 선택한다.

| 작업 규모        | 도구                    | 예시                                |
| ---------------- | ----------------------- | ----------------------------------- |
| 단순 질문/비교   | 메인 대화에서 직접 처리 | "A와 B 차이?", 공식 문서 1~2개 확인 |
| 보고서 수준 조사 | `/research` 스킬        | 3개 이상 기술 비교, ADR 근거 수집   |
| 도메인 전체 작업 | 에이전트                | 설계 + 구현 + 검증이 결합된 작업    |

> **Why:** 서브에이전트 1회 실행 시 하네스 로딩 + 웹 조사로 수천~수만 토큰이 발생한다. 단순 질문에 서브에이전트를 띄우면 메인 대화 대비 2~3배 토큰을 소비한다. 기존 보고서(`research/reports/`)에 이미 조사된 내용은 재조사하지 않고 참조한다.

## 하네스 컨벤션

### 1. Why-First 원칙

규칙만 나열하지 말고 "왜 그런지"를 설명한다. 이유를 이해한 에이전트는 엣지 케이스에서도 올바르게 판단할 수 있다.

```
# BAD
- 컴포넌트 파일명은 PascalCase를 사용한다

# GOOD
- 컴포넌트 파일명은 PascalCase를 사용한다
  → Astro/Svelte 컴포넌트와 일반 유틸 파일을 파일명만으로 즉시 구분하기 위함
```

### 2. Progressive Disclosure

SKILL.md와 가이드 문서는 500줄 이하로 유지한다. 상세 내용은 `references/`로 분리하여 필요할 때만 로드한다. 이는 컨텍스트 윈도우를 효율적으로 사용하기 위함이다.

### 3. Description 공격적 작성

스킬과 에이전트의 description은 트리거 조건을 넓게 작성한다. Claude가 보수적으로 트리거하는 경향이 있으므로, 이를 보상하기 위해 관련 키워드와 상황을 적극적으로 포함한다.

### 4. index.md 기반 탐색

모든 `.claude/` 하위 폴더에는 `index.md`가 있다. 이 파일은 폴더의 역할, 하위 구조, 주요 문서 목록을 설명한다. 에이전트는 새로운 폴더를 탐색할 때 항상 `index.md`부터 읽는다.

## 에이전트 실행 규칙

### 실행 모드

- **에이전트 팀**: 멤버 간 소통이 필요한 복잡한 작업 (Fan-out/Fan-in, Producer-Reviewer 등)
- **서브 에이전트**: 소통 불필요한 독립 하위 작업 (조사, 분석 등)
- 1단계는 에이전트 팀, 2단계는 서브 에이전트로 구성 가능 (중첩 팀 불가)
- 활용 패턴 상세는 `.claude/agents/index.md` 참조

### 작업 전 확인 사항

1. **🚨 브랜치 가드 — 코드 수정·파일 편집 도구(`Edit`/`Write`) 사용 직전 무조건 실행**
   - `git branch --show-current`로 현재 브랜치를 확인한다
   - `main` 또는 `feature/{version}` 버전 루트 브랜치 위라면 **즉시 작업을 중단**하고 사용자에게 버전·페이즈 분할안을 합의받은 뒤 페이즈 브랜치(`feature/{version}-{기능명}`)를 만든다
   - 1개 페이즈로 끝나는 작업도 페이즈 브랜치를 반드시 생성한다 (직접 버전 루트 커밋 금지)
   - 자세한 절차·복구 방법은 `.claude/conventions/guides/workflow.md` §작업 시작 전 필수 가드 참조
2. 이 CLAUDE.md를 읽고 프로젝트 맥락을 파악한다
3. **메모리에서 프로젝트 진행 현황을 확인한다** — 이전 세션에서의 작업 상태, 다음 작업 등을 파악

> **Why:** 브랜치 가드가 다른 확인보다 먼저 와야 하는 이유는, Claude가 "이렇게 수정해줘" 요청에 반사적으로 `Edit`을 호출해 `main`에 변경분이 누적되는 사고가 반복되기 때문이다. 페이즈 단위 PR·롤백·리뷰가 깨지면 워크플로우 전체가 무의미해진다.

아래는 **작업 내용과 관련된 경우에만** 참조한다. 매번 전부 읽지 않는다:

- **의사결정 관련** 작업 시: `.claude/decisions/index.md`
- **기능 구현** 작업 시: `.claude/ia/specs/configDeckIA.md` (필요 시 references/ 하위 문서)
- **코드 작성** 작업 시: `.claude/conventions/index.md` (해당 가이드만 선택 로드)
- **SEO 관련** 작업 시: `.claude/seo/index.md`

### 근거 기반 논의

의사결정이나 기술 논의 시, 반드시 공식 문서나 신뢰도 높은 자료를 근거로 제시한다. 근거 없는 주장이나 추측만으로 의사결정을 유도하지 않는다.

- 기술 비교/추천 시: 공식 문서, 벤치마크, 신뢰할 수 있는 기술 블로그 등의 링크를 함께 제시한다
- ADR 작성 시: 참고 자료(References) 섹션에 근거 링크를 반드시 포함한다
- 컨벤션/가이드라인 작성 시: 해당 규칙의 출처(공식 스타일 가이드, RFC 등)를 명시한다

### 이슈 수정 보고

코드 수정이 필요한 이슈(경고, 에러, 린트 위반 등)를 발견하면 다음 구조로 보고한다. 수정부터 하지 않고 먼저 설명한다.

1. **문제**: 어떤 경고/에러가 발생하는지
2. **원인**: 왜 발생하는지 코드 레벨에서 설명
3. **수정 방안**: 가능한 선택지를 제시하고 각각의 장단점 설명

> **Why:** 단순히 "수정할까요?"만 묻으면 사용자가 맥락을 파악하기 어렵고, 왜 수정이 필요한지 이해 없이 승인하게 된다. 선택지를 제시하면 사용자가 프로젝트 방향에 맞는 판단을 내릴 수 있다.

### 수정 근거 제시

이슈를 파악하고 해결 방법을 제시할 때, 해당 방법을 선택한 **기술적 근거**를 반드시 함께 제시한다. 근거는 반드시 공식 문서, 기술 사양, 또는 검증된 기술 자료를 기반으로 한다.

1. **문제 분석**: 이슈의 원인을 기술적으로 정확하게 파악한다
2. **해결 방법 제시**: 가능한 선택지를 나열한다
3. **근거**: 각 선택지에 대해 공식 문서나 기술 사양을 조사하고, 해당 방법이 적절한 이유를 구체적으로 설명한다. 추측이나 경험적 판단만으로는 부족하다

```text
# BAD: 근거 없이 짧은 설명만 제시
"structuredClone이 안 되니까 JSON.parse(JSON.stringify())로 변경하겠습니다."

# BAD: 공식 문서 없이 경험적 판단만 제시
"보통 이런 경우에는 JSON 방식을 쓰면 됩니다."

# GOOD: 공식 문서 기반의 기술적 근거 제시
"Svelte 5의 $state()는 Proxy 기반 반응성을 사용합니다(Svelte 5 docs: Deep state).
structuredClone은 Proxy 객체를 복제할 수 없어 DataCloneError가 발생합니다
(HTML spec: structuredClone - 복제 불가 타입에 Proxy 포함).

해결 방법:
A) $state.snapshot() — Svelte 5가 제공하는 공식 API로, 프록시를 순수 객체로 변환한다
   (Svelte 5 docs: $state.snapshot)
B) JSON.parse(JSON.stringify()) — 범용적이나 Date, Map, Set 등 특수 타입 손실
C) structuredClone($state.snapshot()) — 특수 타입 보존이 필요한 경우

이 컴포넌트의 데이터는 문자열/불리언/객체만 포함하므로 A가 가장 적절합니다."
```

> **Why:** 기술적 근거 없이 수정하면 잘못된 방향으로 코드가 변경될 수 있다. 공식 문서 기반의 근거를 제시하면 사용자가 해결 방법의 정확성을 검증할 수 있고, 유사한 문제 발생 시 참고 자료로 활용할 수 있다.

### 의사결정 기록

새로운 기술적 의사결정이 발생하면 ADR로 기록한다. `.claude/decisions/template.md`를 기반으로 작성하며, 반드시 참고 자료(공식 문서, 신뢰할 수 있는 블로그 등) 링크를 포함한다.

### 지침 변경 관리

개발 진행이나 논의 과정에서 기존에 명시된 지침, 규칙, 계획과 어긋나는 상황이 발생할 수 있다. 이 경우 다음 절차를 따른다.

1. **기존 지침 확인**: 변경하려는 영역의 관련 ADR, 컨벤션, 가이드라인을 먼저 읽고 현재 규칙이 무엇인지 파악한다
2. **충돌 지점 명시**: 기존 지침의 어떤 부분이 현재 상황과 맞지 않는지 구체적으로 식별한다
3. **변경 논의**: 왜 변경이 필요한지 근거를 제시하고, 사용자와 합의한다
4. **ADR 기록**: 변경 사항을 새로운 ADR로 작성한다. 기존 ADR을 대체하는 경우, 기존 ADR의 상태를 `대체됨`으로 변경하고 새 ADR 번호를 명시한다
5. **관련 문서 동기화**: 변경이 반영되어야 하는 모든 문서(CLAUDE.md, 컨벤션, 가이드라인 등)를 함께 업데이트한다

### 지침 저장 우선순위

새로운 지침이나 규칙이 생길 때, 저장 위치의 우선순위를 따른다.

1. **하네스 문서 먼저**: 코딩 규칙은 `conventions/`, SEO 규칙은 `seo/`, 워크플로우는 `workflow.md` 등 해당 하네스 문서에 먼저 추가한다
2. **메모리는 보조**: 하네스에 저장할 수 없는 정보(사용자 선호, 외부 참조, 프로젝트 진행 맥락 등)만 메모리에 저장한다. 사용자 확인 후 저장한다
3. **중복 금지**: 하네스에 이미 있는 내용을 메모리에 중복 저장하지 않는다

> **Why:** 하네스 문서는 코드와 함께 버전 관리되고, 모든 세션에서 자동으로 로드된다. 메모리는 보조 수단이며, 하네스가 권위 있는 원본(single source of truth)이다.
