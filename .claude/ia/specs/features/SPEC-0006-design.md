# SPEC-0006 UX 설계: Advisory 랜딩 페이지 시리즈

## 0. 설계 전제 및 원칙

### 0.1 이 문서가 다루는 화면

| 화면 | URL 패턴 | 렌더링 |
|------|----------|--------|
| A. 부모 카탈로그 | `/{locale}/advisory` | SSG 정적 |
| B. 자식 advisory 상세 | `/{locale}/advisory/{slug}` | SSG + Svelte 아일랜드 |
| C. 진단기 컴포넌트 | B 페이지 내 삽입 | Svelte 아일랜드 (client:load) |
| D. 페이지 간 이동 / 다국어 / 상태 전환 | 공통 패턴 | 전체 적용 |

### 0.2 설계 원칙

**모바일 퍼스트**: 카드 목록, 필터, advisory 섹션 순서 모두 모바일 기준으로 설계한 뒤 데스크톱으로 확장한다.

**ADR-0019 위계 정신 동일 적용**: `/advisory`(카탈로그)가 개요·진입점, `/advisory/{slug}`(자식)가 실제 콘텐츠와 진단을 담는다. `/ai-config`와 동일한 부모-자식 패턴이므로 기존 컴포넌트 레이아웃 뼈대를 재사용한다.

**단일 진실의 원천 시각화**: frontmatter `affected[].range`가 진단 룰셋이자 영향 범위 표의 원본이다. 진단기 결과와 본문 영향 범위 표가 항상 동일한 데이터를 그려야 한다는 점을 레이아웃에서 명확히 한다.

**접근성 우선 배지 설계**: severity 배지는 색상만으로 의미를 전달하지 않는다. 색상 + 텍스트 레이블 + aria-label을 동시에 제공한다.

## 1. URL 맵 및 정보 구조

### 1.1 URL 맵

| 페이지 역할 | 한국어 URL | 영어 URL | 렌더링 |
|------------|-----------|---------|--------|
| 카탈로그 허브 | `/ko/advisory` | `/en/advisory` | SSG 정적 |
| CVE 상세 (예시) | `/ko/advisory/nextjs-cve-2025-29927` | `/en/advisory/nextjs-cve-2025-29927` | SSG + Svelte 아일랜드 |

ConfigDeck `prefixDefaultLocale: true` 정책에 따라 두 로케일 모두 명시적 prefix를 갖는다.

### 1.2 IA 위계

```
/{locale}
├─ /generator         # 설정 파일 생성기
├─ /ai-config         # AI 도구 설정 카탈로그 (ADR-0019)
├─ /article           # 개발 도구 아티클
└─ /advisory          # ← 이 SPEC (신규)
    ├─ /advisory/nextjs-cve-2025-29927
    └─ /advisory/{slug} ...
```

부모 `/advisory`는 `/article`과 동위 레벨의 독립 콘텐츠 섹션이다. `/ai-config` 섹션과 계층 관계가 아니므로 헤더 네비게이션에서도 독립 항목으로 위치한다.

### 1.3 hreflang 구조

각 advisory 페이지(부모·자식 모두)에 다음 패턴을 적용한다:

```html
<!-- /ko/advisory/{slug} 에서 -->
<link rel="alternate" hreflang="ko" href="https://configdeck.dev/ko/advisory/{slug}" />
<link rel="alternate" hreflang="en" href="https://configdeck.dev/en/advisory/{slug}" />
<link rel="alternate" hreflang="x-default" href="https://configdeck.dev/en/advisory/{slug}" />
```

x-default는 영어 URL로 설정한다. 기존 `/ai-config` 페이지 패턴과 동일하다.

## 2. 화면 A: 부모 카탈로그 `/{locale}/advisory`

### 2.1 설계 근거 (Why-First)

카탈로그 페이지의 목적은 두 가지다: (1) advisory 시리즈가 무엇인지 설명하여 신뢰를 형성하고, (2) 사용자가 자신에게 관련 있는 advisory를 빠르게 찾아 자식 페이지로 진입하게 한다. 검색 유입 사용자는 주로 특정 CVE를 검색한 뒤 자식 페이지로 직접 도달하지만, 카탈로그는 브랜드 앵커(ConfigDeck에 advisory 섹션이 있음을 인식)와 관련 advisory 발견 경로로 기능한다.

필터를 제공하는 이유: advisory가 누적될수록 심각도별, 패키지별 탐색 수요가 생긴다. 초기 1~2건일 때는 필터가 단순하지만, 구조를 처음부터 갖춰야 이후 확장이 일관성을 유지한다.

### 2.4 advisory 카드 정보 위계

카드 내부 정보 우선순위:

1. **severity 배지** — 사용자가 가장 먼저 확인하는 정보. 위험도 판단의 1차 신호
2. **제목** — CVE의 핵심 내용을 한 줄로 표현
3. **CVE ID / GHSA ID** — 검색어와 일치하는 식별자. 검색 유입 사용자의 매칭 확인
4. **패키지명** — 자신의 프로젝트 관련성 판단
5. **publishedAt + 상태** — 최신성과 유효성 확인
6. **CTA** — 자식 페이지로 진입

### 2.5 필터 인터랙션 명세

필터는 정적 페이지에서 URL 파라미터(`?severity=critical&package=next&status=active`)로 구현하거나, Svelte 아일랜드로 클라이언트 필터링으로 구현하는 두 가지 선택지가 있다. advisory 건수가 적은 초기에는 **클라이언트 사이드 필터링(Svelte 아일랜드)**이 더 단순하다. advisory 목록 전체를 정적으로 내려받고, 클라이언트에서 필터·정렬 상태에 따라 렌더링한다.

필터 상태는 URL 파라미터에 반영하여 북마크·공유·브라우저 뒤로가기가 동작하도록 한다.

필터 선택 시 결과가 0건이면 빈 상태(§2.2 참조)를 표시하고, "필터 초기화" 버튼을 제공한다.

### 2.6 빈 상태 (Empty State)

advisory 0건 또는 필터 결과 0건일 때:

```
┌──────────────────────────────────────┐
│                                      │
│       [방패 아이콘 — 중앙]            │
│                                      │
│  현재 조건에 해당하는 advisory가     │
│  없습니다.                           │
│                                      │
│  [필터 초기화]                        │
│                                      │
└──────────────────────────────────────┘
```

## 3. 화면 B: 자식 advisory 페이지 `/{locale}/advisory/{slug}`

### 3.1 설계 근거 (Why-First)

자식 페이지의 사용자 플로우는 SPEC-0006 §3.10에 정의된 5단계를 따른다: 검색 유입 → 헤더 메타 30초 판단 → 진단기 입력 → 패치 안내 → 본 서비스 CTA. 이 플로우가 페이지 섹션 순서와 1:1 대응되어야 한다.

섹션 순서가 콘텐츠 우선인 이유: 검색 유입 사용자는 "내 버전이 영향받는가"를 30초 내에 판단하려 한다. 헤더 메타(심각도, CVSS, 요약)가 최상단에 있어야 이 판단이 가능하다. 진단기가 요약 바로 다음에 오는 이유는, 요약 → 진단 → 결과 → 패치의 선형 플로우가 가장 마찰이 적기 때문이다.

### 3.6 Sunset 배너 설계

상태별 배너는 페이지 최상단(breadcrumb 아래, `<h1>` 위)에 위치한다. 배너 색조와 메시지:

| 상태 | 배너 톤 | 메시지 패턴 | 진단기 |
|------|--------|------------|--------|
| `active` | 없음 | — | 활성 |
| `stale` | 노란 경고 톤 | "이 가이드는 {updatedAt} 기준입니다. 최신 정보는 공식 advisory를 확인하세요." + GHSA 링크 | 활성 |
| `superseded` | 파란 정보 톤 | "이 advisory는 [{후속 slug}]로 대체되었습니다." + 후속 링크 | 비활성 |
| `archived` | 회색 중립 톤 | "이 advisory가 다루는 버전은 EOL(지원 종료)되었습니다." | 비활성(EOL 표시) |

각 배너는 `role="alert"` 또는 `role="note"`를 부여하여 스크린리더에서 상단에 인지될 수 있도록 한다. 색조는 Tailwind의 amber(stale) / blue(superseded) / neutral(archived) 계열로 한다. 경계는 왼쪽 4px 보더 + 배경 tint 패턴.

```
[STALE 배너 예시]
┌────────────────────────────────────────────────────────────┐
│ ▌ 이 가이드는 2025-09-01 기준입니다.                        │
│ ▌ 최신 정보는 공식 advisory를 확인하세요.  [GHSA 원본 →]   │
└────────────────────────────────────────────────────────────┘

[SUPERSEDED 배너 예시]
┌────────────────────────────────────────────────────────────┐
│ ▌ 이 advisory는 후속 [nextjs-cve-2026-xxxxx]로 대체되었습   │
│ ▌ 니다.  [후속 advisory 보기 →]                            │
└────────────────────────────────────────────────────────────┘

[ARCHIVED 배너 예시]
┌────────────────────────────────────────────────────────────┐
│ ▌ 이 advisory가 다루는 Next.js 12.x / 13.x 버전은         │
│ ▌ EOL(지원 종료)되었습니다.                                │
└────────────────────────────────────────────────────────────┘
```

## 4. 화면 C: 진단기 컴포넌트 (AdvisoryDiagnosis — Svelte 아일랜드)

### 4.1 설계 근거 (Why-First)

진단기가 이 서비스의 핵심 차별화 요소다(RES-0004 §1-2). GHSA, Snyk 블로그, NVD 모두 텍스트 가이드만 제공하며, 사용자가 직접 자신의 버전을 semver 범위와 대조해야 한다. 진단기는 이 마찰을 제거한다.

"버전 문자열 입력"을 기본으로 하는 이유: 최소한의 마찰로 즉시 진단이 가능하다. `package.json` 붙여넣기는 더 많은 정보를 주지만 사용자의 추가 행동(파일 열기, 복사)이 필요하므로 옵션으로 제공한다.

진단기를 섹션 5(영향 범위 다음, 패치 안내 이전)에 배치하는 이유: 사용자 플로우의 "영향 여부 확인 → 패치" 단계에 정확히 위치해야 한다. 요약(섹션 2)과 영향 범위(섹션 3) 읽기 후 "내 버전이 이 범위에 해당하는가?"라는 질문이 자연스럽게 발생하는 시점이 섹션 5다.

```
└──────────────────────────────────────────────────────────────────┘
```

비활성 처리 시 입력 필드에 `disabled` 속성을 부여하고, 래퍼에 `aria-disabled="true"`를 마킹한다. 스크린리더가 "이 진단기는 현재 비활성화되어 있습니다"를 인식할 수 있도록 `aria-label`로 보완한다.

## 5. 인터랙션 상태 머신: 진단기 4상태

### 5.1 상태 정의

```
상태 목록:
- IDLE        : 초기 상태. 입력 없음 또는 입력 초기화됨
- AFFECTED    : semver satisfies 결과 true (영향받음)
- SAFE        : semver satisfies 결과 false (영향 없음)
- ERROR       : 입력값이 유효한 semver가 아님
- DISABLED    : status가 archived 또는 superseded (진단 불가)
```

### 5.2 상태 전이 다이어그램

```
                        페이지 로드
                            │
                            ▼
              ┌─────────────────────────┐
              │                         │
   [archived / superseded]       [active / stale]
              │                         │
              ▼                         ▼
          DISABLED                    IDLE
                                       │
                        ┌──────────────┼──────────────┐
                        │              │               │
               [버전 입력]        [package.json     [입력 후
                        │          붙여넣기]        진단하기]
                        │              │               │
                        │         [패키지 추출]        │
                        │              │               │
                        └──────────────┘               │
                                       │               │
                              [진단하기 클릭]           │
                                       │               │
                    ┌──────────────────┼───────────────┤
                    │                  │               │
              [invalid semver]  [satisfies=true]  [satisfies=false]
                    │                  │               │
                    ▼                  ▼               ▼
                  ERROR            AFFECTED           SAFE
                    │                  │               │
                    └──────────────────┴───────────────┘
                                       │
                              [입력값 변경]
                                       │
                                       ▼
                                     IDLE
```

### 5.3 전이 조건 상세

| 현재 상태 | 트리거 | 다음 상태 | 조건 |
|----------|--------|----------|------|
| 페이지 로드 | — | DISABLED | `status === 'archived' \|\| status === 'superseded'` |
| 페이지 로드 | — | IDLE | 그 외 |
| IDLE | "진단하기" 클릭 또는 Enter | ERROR | `!semver.valid(userVersion)` |
| IDLE | "진단하기" 클릭 또는 Enter | AFFECTED | `affected[].range` 중 하나라도 `satisfies(userVersion, range) === true` |
| IDLE | "진단하기" 클릭 또는 Enter | SAFE | 모든 range에 `satisfies === false` |
| AFFECTED / SAFE / ERROR | 입력값 변경 | IDLE | 입력 필드의 input 이벤트 |
| AFFECTED / SAFE / ERROR | "다시 진단" 클릭 | IDLE | 명시적 리셋 |
| IDLE | package.json 붙여넣기 탭 전환 | IDLE | 탭 전환 시 입력값 초기화 |
| IDLE | package.json에서 패키지 추출 성공 | (자동 진단) → AFFECTED/SAFE/ERROR | 추출 직후 자동 진단 |
| IDLE | package.json에 대상 패키지 없음 | IDLE | "패키지 없음" 안내 메시지만 표시, 상태 유지 |

### 5.4 스크린리더 알림 (ARIA Live Region)

진단 결과가 변경될 때(IDLE → AFFECTED/SAFE/ERROR) 스크린리더에 결과를 알린다.

결과 영역에 `aria-live="polite"` 속성을 부여한다. AFFECTED 상태에서는 즉시 인지가 중요하므로 `aria-live="assertive"`로 격상하는 것을 검토한다.

결과 텍스트 예시 (스크린리더 읽기):

- AFFECTED: "진단 결과: 영향받음. next@14.2.20은 CVE-2025-29927의 영향을 받습니다. next@14.2.25로 업그레이드가 필요합니다."
- SAFE: "진단 결과: 안전함. next@15.2.5는 이 취약점의 영향을 받지 않습니다."
- ERROR: "입력 오류: 유효한 버전 형식이 아닙니다."

## 6. 화면 D: 페이지 간 이동 / 다국어 / 상태 전환

### 6.1 breadcrumb 구조

모든 advisory 페이지에 breadcrumb을 표시한다.

부모 카탈로그:

```
Home / Advisory
```

자식 상세:

```
Home / Advisory / {advisory 제목 (줄임 가능)}
```

JSON-LD `BreadcrumbList` 구조: Home → Advisory → {advisory 제목}.

기존 `/ai-config` 자식 페이지(`cursor.astro`)의 `<nav aria-label="Breadcrumb">` + `<ol>` 패턴을 그대로 재사용한다.

### 6.2 hreflang en ↔ ko 전환 UI

기존 ConfigDeck 패턴을 따른다. 헤더 언어 전환 버튼이 현재 페이지의 대응 로케일 URL로 이동한다.

```
[EN]  [KO]  ← 현재 로케일은 강조 표시
```

advisory 페이지에서 언어 전환 시 동일한 slug의 대응 로케일 URL로 이동한다:

- `/ko/advisory/nextjs-cve-2025-29927` ↔ `/en/advisory/nextjs-cve-2025-29927`

두 로케일 파일이 모두 존재해야 hreflang 태그가 활성화된다. 한 쪽만 존재하는 경우 hreflang 상호 참조를 생략한다.

### 6.3 archived/superseded 상태에서 진단기 영역 처리

§4.3에서 컴포넌트 레벨 처리를 정의했다. 페이지 레벨에서 추가로:

- `superseded` 상태: `<head>`에 `rel=canonical`을 후속 페이지 URL로 설정한다 (SPEC-0006 §3.8, Google Search Central noindex 금지 원칙 준수).
- `archived` 상태: canonical은 자기 자신 URL을 유지한다. sitemap에서 제외하지 않는다.
- `stale` 상태: canonical 변경 없음. 배너만 표시.

## 7. 반응형 전략 요약

### 7.1 브레이크포인트 정책

기존 ConfigDeck 패턴(`lg:` = 1024px)을 따른다.

| 구간 | 적용 레이아웃 |
|------|------------|
| < 768px (모바일) | 단일 컬럼, 필터 가로 스크롤 칩, 카드 전체 너비 |
| 768px ~ 1023px (태블릿) | 단일 컬럼, 일부 카드 2열(`sm:grid-cols-2`) |
| 1024px+ (데스크톱) | advisory 카탈로그: 카드 2열(`lg:grid-cols-2`). 자식 페이지: 단일 컬럼 좌우 여백 확대 |

### 7.2 카탈로그 페이지 반응형

- 모바일: 필터 칩이 가로 스크롤(`overflow-x-auto`) + 스냅. 카드 단일 컬럼
- 데스크톱: 필터 칩 줄바꿈 허용. 카드 2열 그리드

필터 UI 상세: 모바일에서는 심각도·패키지·상태 필터를 각각 별도 행의 가로 스크롤 칩으로 배치한다. 세 행이 세로로 쌓여 세 줄의 칩 행이 된다. 각 행은 레이블(심각도:, 패키지:, 상태:)로 구분한다.

### 7.3 자식 advisory 페이지 반응형

자식 페이지는 콘텐츠 중심 단일 컬럼 레이아웃이다. `max-w-4xl` 또는 `max-w-3xl` 중앙 정렬. 섹션 순서는 모바일과 데스크톱 동일.

영향 범위 표(§3.2 섹션 3)는 모바일에서 가로 스크롤이 가능하도록 `overflow-x-auto` 래퍼를 적용한다.

### 7.4 진단기 반응형

진단기 컴포넌트는 페이지 전체 너비(`max-w` 상속)에서 동작한다. 모바일/데스크톱 레이아웃 차이 없이 단일 컬럼.

결과 카드(AFFECTED/SAFE/ERROR)는 컴포넌트 너비에 맞춰 전체 너비로 표시한다.

## 8. 접근성 고려사항

### 8.1 severity 배지

severity 배지는 색상만으로 의미를 전달하지 않는다. 반드시 색상 + 텍스트 레이블 + aria-label을 동시에 제공한다.

```html
<span
  class="bg-red-100 text-red-800 ..."
  aria-label="심각도: Critical (9.1)"
>
  CRITICAL
</span>
```

심각도 레이블은 영어로 표기한다(`CRITICAL`, `HIGH`, `MEDIUM`, `LOW`). 이는 국제적으로 통용되는 표준 레이블이므로 번역하지 않는다. 다만 `aria-label`에는 로케일에 따라 "심각도: Critical" (한국어) 또는 "Severity: Critical" (영어) 형식으로 보완한다.

색조 매핑:

| severity | 배경 톤 | 텍스트 톤 |
|---------|--------|----------|
| critical | 빨간 | 빨간 어두운 |
| high | 주황 | 주황 어두운 |
| medium | 노란 | 노란 어두운 |
| low | 파란 | 파란 어두운 |

### 8.2 진단 결과 스크린리더 알림

§5.4에서 정의한 `aria-live` 영역을 통해 결과 변경을 알린다.

결과 영역을 포커스 이동 없이 업데이트하므로, 스크린리더 사용자가 "진단하기" 버튼 활성화 후 결과를 즉시 인지할 수 있어야 한다.

AFFECTED 결과는 즉각적 조치가 필요한 정보이므로 `aria-live="assertive"` 사용을 우선 검토한다. 단, assertive는 사용자 현재 읽기를 중단하므로 결과 텍스트가 간결하고 명확해야 한다.

### 8.3 진단기 비활성 상태 (archived/superseded)

비활성 입력 필드에 `disabled` 속성 부여. 비활성 이유를 설명하는 텍스트를 시각적으로 노출하고, `aria-describedby`로 연결한다.

### 8.4 외부 링크 처리

모든 외부 링크("GHSA 원본 →", "NVD →", "공식 advisory에서 확인 →")는:

- `target="_blank" rel="noopener noreferrer"` 적용
- 링크 텍스트에 "(새 탭에서 열림)" 또는 시각적 아이콘(↗) 제공
- 스크린리더를 위해 `aria-label`에 "(새 탭)"을 추가하거나 시각적으로 표시된 아이콘에 `aria-hidden="true"` + 스크린리더 전용 텍스트(`.sr-only`)를 함께 제공

### 8.5 코드 블록 접근성

패치 명령어 코드 블록은 `<pre><code>` 마크업을 사용한다. 복사 버튼은 `aria-label="npm install next@14.2.25 복사"` 형식으로 구체적인 내용을 포함한다.

## 11. ui-publisher 인수인계: 시안 작성 시 주의 항목

다음 7개 항목을 시안 작성 전에 반드시 확인한다.

1. **진단기 Svelte 아일랜드 마운트 위치**: `AdvisoryDiagnosis` 컴포넌트는 Astro 페이지에서 `client:load` 디렉티브로 삽입한다. Astro SSG 빌드 시 컴포넌트의 `affected` 배열은 `getEntry().data.affected`에서 읽어 props로 전달한다. 빌드 타임에 전달하고 클라이언트에서 진단 로직만 실행한다.

2. **severity 배지 색상 + 텍스트 동시 표기**: 색상만으로 심각도를 표현하지 않는다. `SeverityBadge` 컴포넌트에 `aria-label`을 반드시 포함한다. WCAG 1.4.1(색상 단독 정보 전달 금지) 준수.

3. **sunset 배너 위치**: breadcrumb `<nav>` 바로 아래, `<h1>` 바로 위에 배치한다. `active` 상태일 때 배너 자체가 렌더링되지 않아야 하며(HTML에 빈 div 잔존 금지), Astro 조건부 렌더링으로 처리한다.

4. **면책조항 상시 표기**: 진단기 컴포넌트 결과 영역 하단에 면책조항이 항상 표시되어야 한다. IDLE 상태에서도 표시. 결과 상태(AFFECTED/SAFE/ERROR)에서 면책조항이 결과 카드 아래에 위치한다. 면책조항을 숨기거나 토글하지 않는다.

5. **필터 컴포넌트 URL 파라미터 동기화**: `AdvisoryFilter` Svelte 컴포넌트가 필터 상태를 URL 파라미터로 동기화할 때, `history.replaceState`를 사용한다(새 히스토리 항목 추가 금지). 브라우저 뒤로가기 시 필터 상태가 복원되어야 한다.

6. **영향 범위 표의 모바일 가로 스크롤**: 섹션 3의 `<table>`은 `overflow-x-auto` 래퍼 `<div>` 안에 배치한다. 표 자체에 `min-width`를 지정하여 모바일에서 컬럼이 압축되지 않도록 한다. `<table>` 요소에 `role`을 별도로 부여하지 않으며, `<caption>` 또는 `aria-labelledby`로 테이블 목적을 명시한다.

7. **`superseded` 상태의 canonical 처리**: `status === 'superseded'`인 advisory의 Astro 페이지에서는 `<link rel="canonical">` 태그를 `Layout.astro`에 넘기는 `canonicalUrl` prop(또는 동등한 메커니즘)을 통해 후속 advisory URL로 설정한다. `Layout.astro`가 현재 이 prop을 지원하는지 먼저 확인하고, 없으면 `Layout.astro` 수정이 선행 작업이 된다.
