# features/

기능 단위 기획서(SPEC)를 보관하는 폴더이다. 서비스 전체 IA는 상위 `specs/configDeckIA.md`에서 관리하고, 개별 기능의 상세 기획은 이곳에 `SPEC-{번호}` 형식으로 기록한다.

## 이 폴더의 역할

- **기능 기획의 단일 소스(Single Source of Truth)**: 기능의 배경, 목표, 설계, 실행 계획을 한 문서에 모은다
- **ADR과의 연결**: 기술적 의사결정은 ADR에, 기능의 사용자 가치와 실행 계획은 SPEC에 기록한다. 두 문서는 서로를 상호 참조한다
- **에이전트 작업의 기준점**: 기능 구현 에이전트는 이 폴더의 SPEC 문서를 참조하여 작업 범위와 확인 지점을 파악한다

## 파일 명명 규칙

- 형식: `SPEC-{4자리 번호}-{kebab-case-slug}.md`
- 예시: `SPEC-0001-option-generator-enhancement.md`
- 번호는 순차 증가. 폐기된 기획서의 번호는 재사용하지 않는다

## 작성 규칙

1. **템플릿 사용**: 새 기획서는 반드시 [`../../templates/feature-spec.md`](../../templates/feature-spec.md)를 복사해 시작한다
2. **Why-First**: CLAUDE.md의 Why-First 원칙을 따른다. "무엇을"뿐 아니라 "왜"를 반드시 포함한다
3. **근거 기반**: 설계 근거는 공식 문서, 벤치마크, 레퍼런스 링크를 포함한다
4. **ADR 연결**: 기술적 의사결정이 포함된 기획은 별도 ADR을 작성하고 frontmatter의 `related_adrs`에 명시한다
5. **상태 관리**: frontmatter의 `status`를 진행 상황에 맞게 업데이트한다
6. **변경 이력**: 주요 변경 시 문서 하단 Changelog에 기록한다

## 상태(status) 정의

| 상태    | 의미                                             |
|---------|--------------------------------------------------|
| 초안    | 작성 중. 아직 검토 요청 전                       |
| 검토 중 | 사용자/이해관계자 검토 대기                      |
| 승인됨  | 검토 완료, 구현 착수 가능                        |
| 구현 중 | 실제 구현 작업 진행 중                           |
| 완료    | 구현 및 검증 완료                                |
| 보류    | 일시 중단                                        |
| 폐기    | 폐기됨. 다른 기획으로 대체되었거나 방향이 바뀜   |

기획서 작성/업데이트는 [product-planner](../../../agents/product-planner.md) 에이전트에 위임한다. 에이전트는 이 index와 템플릿을 따라 일관된 형식의 기획서를 생성한다.
