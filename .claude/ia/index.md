# ia/

프로젝트의 정보 구조(Information Architecture) 및 기획 문서를 관리하는 폴더이다.

## 이 폴더의 역할

- 서비스 기획서, IA, 페이지 구조, 사용자 플로우 등 기획 단계의 문서를 보관한다
- 에이전트는 기능 개발 시 이 폴더의 기획 문서를 참조하여 서비스의 의도와 범위를 파악한다

## 폴더 구조

- `index.md` — 이 파일. 폴더 역할 설명 및 기획 문서 목록
- `specs/` — 서비스 전체 IA 및 기능별 기획서(SPEC)
  - `configDeckIA.md` — 서비스 전체 IA
  - `references/` — IA의 상세 참조 문서 (페이지 구조, 엣지 케이스 등)
  - `features/` — 기능 단위 기획서(SPEC-{번호})
- `templates/` — 기획 문서 템플릿
  - `feature-spec.md` — 기능 기획서 템플릿

## 문서 구분

| 구분              | 위치                            | 용도                                           |
|-------------------|---------------------------------|------------------------------------------------|
| 서비스 IA         | `specs/configDeckIA.md`         | 서비스 전체 정보 구조, 페이지 목록, 공통 규칙  |
| 기능 기획서(SPEC) | `specs/features/SPEC-{번호}.md` | 개별 기능의 배경/목표/설계/실행 계획           |
| 참조 문서         | `specs/references/`             | IA의 세부 상세 (페이지 구조, 엣지 케이스 등)   |
| 템플릿            | `templates/`                    | 새 문서 작성 시 복사해 사용하는 틀             |

> **Why SPEC 분리:** 서비스 전체 IA는 변동이 적은 반면, 기능 기획은 수시로 추가·변경된다. 동일 파일에 누적하면 파일이 비대해지고 변경 이력이 섞여 추적이 어렵다. 기능 단위로 파일을 분리하면 Git 이력, 리뷰 범위, 에이전트 참조가 모두 명확해진다.

## 기획서 작성 규칙

1. **새 기획은 SPEC로 시작**: 새 기능/보강 작업은 [`templates/feature-spec.md`](templates/feature-spec.md)를 복사해 `specs/features/SPEC-{번호}-{slug}.md`로 작성한다
2. **ADR과의 역할 분리**:
   - **SPEC**: 사용자 가치, 기능 범위, UX, 실행 계획
   - **ADR**: 기술적 의사결정 (아키텍처, 라이브러리 선택, 구조 변경)
   - 기술적 결정이 포함된 SPEC은 별도 ADR을 작성하고 양쪽에서 상호 참조한다
3. **전담 에이전트**: 기획서 작성/업데이트는 [product-planner](../agents/product-planner.md) 에이전트를 활용한다
